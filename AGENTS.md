# AGENTS.md

Guidance for AI coding agents working in this repository.

## Project overview

Single-file NetSuite SuiteScript 2.1 Suitelet (`ao-ns-dashboard.js`) deployed inside a NetSuite account to extend it with admin/automation tooling: detect record type from internal ID, view full record + sublist details, look up arbitrary record/sublist fields, and edit / mass-create / mass-save / mass-delete records in bulk.

There is no build step, no package manager, no test suite. The "deployment" is uploading the JS file to NetSuite as described in `README.md` (Customization → Scripting → Scripts → New, then create a Script Record + Deployment).

## Architecture

Everything lives inside a single AMD `define([...], function(record, search, query, runtime) { ... })` module exporting `{ onRequest: main }`. The Suitelet is both server- and client-rendered: the server function returns HTML strings that include inline `<script>` blocks, and the same Suitelet URL handles JSON POST callbacks from those scripts.

### Request dispatch (`main`, ~line 96)

The Suitelet uses two query-string parameters to multiplex behaviour:

- `page=<name>` — selects which page to render. Defaults to `welcome`.
- `command=<name>` — when present, the request is treated as a JSON-body POST callback; the response is plain text written via `context.response.write`, not HTML.

Pages register themselves into the module-level `pages` object:

```js
pages["edit-records"] = {
  label: "Edit Records",
  render: editRecordsPage,                // (context) => htmlString
  commands: { [commandName]: handler }    // (context) => stringOrJsonString
};
```

`main` looks up `pages[pageParam].render(context)` for HTML requests, and for command requests it scans every page's `commands` map for the matching handler. The left-nav drawer is generated by iterating `Object.keys(pages)`. **Adding a new page means appending another `pages[...] = { ... }` block** — there's no central registry to update beyond that.

### Page list (in file order)

1. `welcome` — landing page (`welcomePage`, ~line 194)
2. `record-type` — detect record type from internal/external ID (`typePage` + `handleTypeListing`, ~line 218)
3. `record-details` — load + display all fields and every sublist for a single record (`detailsPage` + `detailsListing`, ~line 352)
4. `lookup-fields` — bulk lookup of fields, including nested sublist/subrecord paths (`lookupFieldsPage` + `handleLookupFields` + `pathLookupFields`, ~line 595)
5. `edit-records` — bulk `set` / `insert` / `remove` actions on records and sublists (`editRecordsPage` + `handleEditRecord` + `handleEditRecordAction`, ~line 812)
6. `create-records` — mass record creation (~line 1353)
7. `mass-save` — re-save records to trigger workflows (~line 1538)
8. `mass-delete` — mass deletion (~line 1600)

### Bulk-task UI pattern

Pages 4–8 all share the same client-side runner, generated by helpers near the bottom of the file:

- `runCommandJs()` — defines `model`, `runCommand(batch)`, `pageStart`, `pageCount`.
- `runNextJs()` — picks the next un-started task (or batch sharing the same `group`) and POSTs it to the page's command URL.
- `runAllJs()` — parses the textarea, populates `model`, then drives `runNext`.
- `renderStatusJs()` — re-renders the progress table and powers the CSV download.
- `taskListAndRunStatusJs(label)` — emits the textarea + Run/Download UI.

Each page composes these into its own `<script>` block, sets `commandPostUrl`, optionally pushes a `modelProcessor` to assign tasks to groups (e.g. `edit-records` groups by record type + ID so multiple edits to the same record are sent in one POST and saved together), and calls `taskListAndRunStatusJs(...)`. The server handler receives a JSON array of task strings and must return either a single string or a JSON array of strings of the same length.

### Task input format

Task lines are pipe-delimited: `Record Type|Internal ID|Location|Field Values|Action`. Within those:

- **Field Values** — `&`-separated `fieldId=value` pairs. Use `\&` to escape.
- **Location** (sublist path) — `/`-separated. Use `\/` to escape.
- **Pipe** — use `\|` to escape inside any segment.

The three `splitVerticalBar` / `splitAmpersand` / `splitSlash` helpers (~line 1698) implement this escape handling via random-sentinel substitution. Use them rather than calling `String.split` directly.

### Record-type resolution

`allRecordTypes` is built once per request from `record.Type` plus a small `undocumentedRecordTypes` map (currently `TRANSFER` → `transfer`, `CURRENCY_REVALUATION` → `fxreval`). `getRecordType(input)` (~line 76) accepts the `record.Type` key, the underlying string value, or a fuzzy version with non-letters stripped. When adding support for new undocumented record types, add them to that map rather than hard-coding strings at call sites.

### Other conventions

- `version` constant (~line 34) is shown in the header — bump it when shipping a change worth surfacing to users.
- The header turns red on non-sandbox environments (detected from hostname containing `-sb`) — see ~line 147.
- External CSS/JS is loaded from cdnjs.cloudflare.com (Material Design Lite, jQuery, Select2). Earlier code used `getmdl.io`; the migration to cloudflare is recent (commit `5dde223`).
- Indentation in this file is **tabs**. Preserve that exactly when editing — mixing tabs and spaces inside the template literals will show up in the rendered HTML.
- The whole module is one giant function, so there's no import/export — helpers are just function declarations within the closure. Add new helpers near related code rather than at the bottom unless they're truly generic (alongside `normalizeKey`, `splitAmpersand`, etc.).

## Working in this repo

- There is no lint, build, or test command. Verification is manual: upload to a NetSuite sandbox and exercise the relevant page.
- `.idea/` is git-ignored; the JS file itself is the entire deliverable.
